1. Оценка спецификации
Спецификация очень сильная по содержанию:
Закрывает не только алгоритмы, но и инфраструктуру (CDN/WAF → Gateway → App).
Даёт best practices по IP (trusted proxies, заголовки, нормализация, отсутствие порта).
Описывает разные алгоритмы (Fixed/Sliding/Token/Leaky) и прямо даёт production Lua‑скрипт для Sliding Window Log.
Есть бизнес‑логика (IP+login, уровни severity, блокировки, 2FA), HTTP‑контракт (RateLimit‑/Retry‑After), observability, конфигурирование, feature flags, тестирование и чек‑лист.
То есть как “эталонный конспект” для прод‑rate‑лимитера — очень хорошо.
2. Что у тебя уже есть в pessimistic‑реализации
Сейчас в PessimisticLimiter и ClientIP у тебя реализовано:
Алгоритм: Sliding Window Log на ZSET per IP (rate:{ip}) с очисткой старых записей и подсчётом в окне.
Техника concurrency: пессимистический lock lock:rate:{ip} через SET NX EX, с ретраями, затем ZREM… → ZCOUNT → ZADD внутри критической секции.
Ключ без порта: ClientIP отрезает порт от RemoteAddr, так что ephemeral port не ломает лимит.
Это уже корректная учебная реализация пессимистичной стратегии.
3. Чего не хватает до “best practices” именно для pessimistic стратегии
Ниже — по сути “diff” между текущим кодом и тем, что ты сам описал в specification.md (разделы 1, 2, 3, 4, 5, 8), применительно к пессимистичному варианту.
3.1. Извлечение IP и формирование ключа
Сейчас:
ClientIP:
если есть X-Forwarded-For — возвращает строку как есть (может быть список IP через запятую, без нормализации);
иначе X-Real-IP;
иначе RemoteAddr с отброшенным портом.
В спецификации:
Явный список доверенных прокси (CIDR) и проверка, что мы доверяем заголовкам только от них.
Разбор X-Forwarded-For и взятие первого реального IP, нормализация (IPv4/IPv6), отсутствие порта.
Композитные ключи ratelimit:login:ip:... и т.п., префикс окружения (prod:…).
Чего не хватает:
Учесть trusted proxies и корректно парсить X-Forwarded-For, а не брать строку целиком.
Добавить нормализацию IP и, по желанию, префикс окружения для ключей.
Для anti‑bruteforce логина — завести отдельные ключи (по IP, по логину, по комбинации), а не только rate:{ip}.
3.2. Собственно lock (distributed locking)
Сейчас:
SET ... NX EX 1 с lockValue = now.UnixNano(), но при DEL значение не проверяется (любой владелец может удалить любой лок).
TTL фиксированный (1 секунда), без явной связи с максимальным временем обработки.
При неудачных ретраях возвращается ошибка → дальше handler решает, что делать.
Best practices для пессимистичных локов:
Хранить уникальное значение в ключе (uuid) и при DEL сверять его (через Lua‑скрипт), чтобы не удалить чужой lock.
TTL подбирать так, чтобы минимизировать deadlock и “висячие” локи, но и не истекать слишком рано.
При невозможности взять lock / ошибках Redis — действовать по политике fail‑safe (обычно блокировать запрос, а не пропускать).
Чего не хватает:
Безопасного снятия локов (проверка lockValue в DEL через Lua).
Чётко задокументированной политики: при невозможности взять лок или ошибке Redis — либо блокируем (429/500), либо пропускаем (обычно это плохо). Сейчас поведение, вероятно, “500”, но не задекларировано как fail‑safe/fail‑open.
3.3. HTTP‑контракт и поведение на превышении лимита
Сейчас (по проекту в целом):
Есть простая семантика Allow/deny, но нет полностью реализованного контракта c RateLimit-* и Retry-After как в specification.md (раздел 4.2).
Best practice:
Возвращать:
429 Too Many Requests;
заголовки X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset, Retry-After;
JSON‑тело с пояснением.
Чего не хватает:
Единообразного middleware / helper’а, который от имени всех стратегий пишет стандартные rate‑limit заголовки и тело ответа.
3.4. Observability (метрики, логи)
Сейчас:
Спецификация описывает Prometheus‑метрики, лог‑структуры и алерты, но в Go‑коде pessimistic‑лимитера этого нет.
Best practice:
Метрики:
сколько раз сработал лимит (per strategy, per endpoint, per key‑type);
сколько раз были проблемы с lock (неудачные acquire, таймауты);
latency самого Allow.
Логи:
структурированное логирование превышений лимита и аномалий (как в RateLimitLog).
Алерты: рост ratelimit_exceeded_total, рост latency Redis и т.п.
Чего не хватает:
Интеграции с Prometheus / логгером: хотя бы счётчики exceeded, lock_acquire_failed, гистограмма check_duration.
Базовых алертов (это уже поверх метрик, не в коде).
3.5. Конфигурируемость
Сейчас:
Константы lockTTL, maxLockRetries, lockRetryDelay, limit, windowSec заданы в коде/задании.
Best practice (раздел 6):
Лимиты и параметры лока задаются через конфигурацию:
per endpoint;
per key‑pattern (IP / login / API‑key);
без перекомпиляции/деплоя.
Чего не хватает:
Вынести:
размеры окна и лимиты;
параметры lock (TTL, retry‑policy);
настройки ключей
в конфиг (YAML/env) и/или динамические флаги.
4. Краткий план по остальным стратегиям (в рамках этой спецификации)
Ниже — “скелет” плана, если ты захочешь довести весь набор стратегий до вида “учебный код, близкий к прод”.
4.1. Naive (in‑memory)
Цель: оставить как чисто демонстрационный вариант с чётким ярлыком “НЕ использовать в проде”.
План:
Убедиться, что в README/subtask‑1 явно написано “demo only, показывает гонки и отсутствие масштабируемости”.
Добавить (если нужно) простой unit‑/load‑тест, который демонстрирует “пролезание” сверх лимита или несогласованность.
4.2. Pessimistic (что делать дальше)
На основе текущей реализации:
Довести ClientIP/ключи до spec (trusted proxies, нормализация, без порта, префикс окружения).
Усилить лок:
хранить uuid в значении,
снимать лок через Lua‑скрипт DEL IF value == expected.
Вынести параметры lock/лимитов в конфиг.
Добавить:
rate‑limit заголовки,
метрики (exceeded/locks/errors),
структурированные логи.
4.3. Optimistic (Redis WATCH)
Алгоритм по strategies-overview.md:
WATCH rate:{ip};
читаем ZSET (ZRANGEBYSCORE/ZCOUNT) после очистки старых записей;
если превышено — UNWATCH и возвращаем “deny”;
иначе MULTI { ZADD ... } EXEC;
если EXEC вернул nil → был конфликт → повторить.
План:
Реализовать OptimisticLimiter с использованием WATCH / MULTI / EXEC вокруг ZSET.
Использовать тот же слой IP/ключей и HTTP‑контракт, что и pessimistic/atomic.
Ограничить максимальное число ретраев и считать их в метриках.
4.4. Atomic (Redis Lua, Sliding Window Log)
Это “цель” из спецификации.
План:
Вынести production‑Lua скрипт из specification.md (раздел 3.2) как строку в Go‑коде и обёртку, которая:
делает EVALSHA/EVAL по ключу;
возвращает структуру {allowed, current, limit, retry_after, now}.
AtomicLimiter будет:
строить правильный ключ (по IP / login / комбинированный);
вызывать Lua и по его результату:
либо пропускать запрос,
либо возвращать 429 с заполненными RateLimit‑заголовками и Retry-After.
Подключить те же метрики/логи, что и для других стратегий.
Сделать нагрузочный тест (как в scripts/load-test) наравне с pessimistic/optimistic.
